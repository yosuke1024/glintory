# System Instructions for AI Agent: Architecture & Coding Conventions (Glintory)

You are an autonomous, self-verifying AI Developer Agent. When developing and refactoring this application, you MUST strictly adhere to the following principles.

---

## 1. Core Architecture (アーキテクチャ原則)
* **Layered & Domain-Driven Structure:** 仕様書に定義された階層構造（`domain/`, `collectors/`, `services/`, `infrastructure/`, `web/`, `ai/`）に厳密に従え。
* **Rule of Three / WET over DRY:** コンポーネント間の過度な結合による「依存関係の迷子」を避けるため、2回までのコード重複（WET）は積極的に許容する。しかし、「3箇所以上での完全な重複」が発生した場合のみ、システム的な保守性を考慮し共通化（DRY）を検討すること。
* **Single Source of Truth (SSOT):** DBスキーマやドメインモデル、各種インターフェース（`Collector` 等）を唯一の情報源とし、依存関係を整理せよ。
* **File Size Limit:** 1つのPythonファイルに機能を集中させず、400行以内に保つこと。

## 2. API & State Management (外部連携と永続化)
* **State and Transaction Boundaries:** 重要な状態遷移や履歴保存は、同一のSQLiteトランザクション内で行うこと。
* **Intent/State Pattern:** 外部API（GitHub/Hacker News等）の収集処理はデータベーストランザクションの「外」で行い、完了したデータをトランザクションを用いて永続化せよ。

## 3. Coding Standards (実装規約)
* **Strict Public Typing & Local Inference:** パブリックなインターフェース（関数の引数・戻り値、APIレスポンス、クラスプロパティ等）には、厳密な型ヒント（TypeScript/Python Type Hints, Pydantic）を必須とする。関数内のローカル変数は、トークン節約と可読性向上のため、安全な型推論を積極的に利用せよ。
* **Explicit Error Handling:** エラーの握り潰し（サイレントフェイル）は厳禁。エラーは適切にログ出力し、Root Causeを自律特定・修復可能にせよ。特にCollectorのエラーはSource Healthに記録すること。
* **Language Rules:** コードおよびコメント、コミットメッセージは英語で記述せよ。ただし、ユーザー（人間）とのチャットコミュニケーションは日本語で行うこと。

## 4. Self-Verification (自己検証プロセス) & TDD Rules
* **Atomic Steps:** 1つの機能追加・修正ごとに、必ず単体テストによる動作検証を行い、成功を確認してから次のステップ（機能実装）に進むこと。
* **TDD (Test-Driven Development) Cycle:** 全ての新機能、ビジネスロジック、およびバグ修正を行う際、以下の厳密なTDDサイクルを遵守せよ：
  1. **Red (テスト失敗状態):** 実装コードを書く前に、仕様書に沿ったテストコードを作成し、実行して「失敗する」ことを確認する。
  2. **Green (テスト通過状態):** テストを通過させるための最小限の実装コードを書く。
  3. **Refactor (リファクタリング):** テストがGreenであることを維持したまま、コードの重複排除やアーキテクチャ規約に沿ったリファクタリングを行う。
* **Fake Adapters / HTML Fixtures:** 外部依存（GitHub API, Hacker News API, RSSフィード）がある場合は、最小限の保存したfixtureを使い、パーサーやコレクターの契約をオフラインで高速かつ決定論的にテスト可能にせよ。

## 5. Security & Environment (セキュリティと環境設定)
* **Strict Env & Secrets Masking:** 認証キー等は環境変数（`.env`）から読み込み、Pydantic Settings等によるバリデーションを実行せよ。`.env`のコミットは禁止し、エラーログプロセスで機密情報をマスキングせよ。
* **Injection & Dynamic Execution Prevention:** DBクエリやシェル呼び出しを実行する際は、SQLAlchemyプリペアードステートメントやサニタイズを徹底せよ。また、悪意あるユーザー入力による任意のコード実行を防ぐため、`eval`、`exec`、入力値依存の動的インポートは原則禁止とする。
* **SSRF Prevention:** ユーザーが指定した任意のURLにアクセスする際は、SSRF対策を徹底せよ。`file://`のアクセス拒否、localhost、link-local、private network宛てのURLアクセスを拒否し、リダイレクト後の宛先も検査すること。
* **Dependency Safety:** 新しいパッケージを導入する際は、必ず事前に脆弱性チェックを行うこと。

## 6. Performance & Protective Design (防御的パフォーマンス設計)
* **Idempotency First:** すべての書き込み処理およびジョブ実行は、重複して呼び出されても安全（冪等）でなければならない（例：同一URL of duplicated registration prevention, Signal件数の重複増の防止）。
* **SQLite Best Practices:** SQLite初期化時は必ず WAL モードの有効化、外部キー制約の有効化、ビジータイムアウトの設定を行え。
  ```sql
  PRAGMA journal_mode = WAL;
  PRAGMA foreign_keys = ON;
  PRAGMA busy_timeout = 5000;
  ```

## 7. Observability (可観測性とデバッグ容易性)
* **Traceable Context Logging:** エラー単体ではなく「処理の開始から完了まで」を追跡可能な Correlation ID を付与し、構造化ログを出力せよ。

## 8. Agent Collaboration Rules (AI行動ルールと例外基準)
* **Component-Level Modification:** 既存のファイルを修正する際、ファイル全体を丸ごと再生成または上書きするような変更は避けること。必ずツールを駆使し、修正対象の行や関数にスコープを当てた「部分限定修正」を行うこと。
* **Document the "Why" over "What":** 特殊な仕様、ハック、または通常のセオリーから外れた実装には、必ず「なぜ（Why）そうしたか」の背景をコード内に直書きせよ。
* **Limit Blast Radius:** タスク対象外 of code で古い実装や気付いた改善点があっても「ついで修正」は厳禁。影響範囲が想定以上に広くなる場合は、事前にPlan（計画）を作成しUserの承認を得よ。
* **Documentation Updates:** 新機能の追加や仕様変更を伴うコーディングタスクを完了した際は、実装コードだけでなくドキュメント類のアップデートも自発的に行うこと。

## 9. Spec-Driven Implementation
* **Single Source of Truth:** 実装に着手する際、エージェントはチャット履歴の断片的な会話に依存せず、常に仕様書（`Glintory_PRODUCT_SPEC.md` 等）を唯一の情報源（正）として扱うこと。
* **Spec vs Code Conflict:** 実装中に仕様書と既存コードが矛盾する場合、仕様書を正とする。技術的な理由で仕様通りに作れない、または重大なリスクが見つかった場合は、独断でコードを変更せずユーザーに確認を仰ぐこと。
