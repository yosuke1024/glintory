## What happens `FactIndexer.index_fact()` only trains the vocabulary when the facts codebase has no documents at all: ```python # Ensure GlobalVocabulary is trained if self.global_vocab.get_codebase_doc_count(FACTS_CODEBASE_ID) == 0: await self._train_vocabulary() await self._index_fact(fact) ``` Once the vocabulary has been trained even once, which is the normal steady state, every subsequent `index_fact()` call skips training entirely. The fact is embedded and upserted, but its tokens are never registered with `GlobalVocabulary` and the facts document count is never incremented. Two consequences follow. A fact containing terms not already in the vocabulary gets a sparse vector that simply omits those terms, so sparse retrieval cannot match on them. And because the document count and document frequencies do not move, the IDF statistics for the facts corpus grow progressively staler as facts accumulate, degrading hybrid ranking for every fact. Dense retrieval is unaffected, so facts indexed this way are still findable. The degradation is confined to the sparse half of the hybrid search and to ranking quality. ## Where `src/vector_core/facts/indexer.py`, in `index_fact()` (around line 329). ## Why it matters now Both single-fact code paths in mcp-notes route through `index_fact()`: `update_fact` has done so since mcp-notes#40, and `add_fact` now does too (mcp-notes#70). So this is the path taken by the two most common fact operations, and the vocabulary drifts further out of date with every one of them. The bulk path is unaffected: `index_all()` uses the two-pass pattern and calls `register_codebase()`, which rebuilds the facts contribution correctly. ## Suggested fix Incrementally update the vocabulary in `index_fact()` rather than skipping it, using `update_codebase_incremental()` with the new fact's token set and a document-count delta, in the same way the codesearch incremental indexer maintains its contribution.
